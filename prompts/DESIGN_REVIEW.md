# 新 prompt 设计：审视结论

把游戏描述从"投喂材料"升级为"结构化 prompt 架构"。核心是一条原则：
**prompt 必须与 rubric 对齐，但"对齐"不等于"把检查项抄进 prompt"。**
按维度区别对待。

## 一、功能维 (D1 + D2)：显式且完整地写
- 让模型知道"要能跑、要处理关窗、要实现这些游戏规则"是**任务本身**，
  不是作弊。这部分写全，模型照做自然过 D1/D2。
- 但用**行为语言**描述，**不抄评估器内部**：
  - 写"正确处理输入事件、程序能干净退出、不要失控的忙等循环"，
  - 不写"我们会检测 `event.get()`/`event.poll()` 和 `while True: pass`"。
  后者是把考卷答案递给模型。
- D2 的 checkpoint 必须写全（那就是功能规格），措辞为"必须实现以下功能"，
  不是"我们会按 0/1/2 打分"。

## 二、质量维 (D3 + D4)：按表格顺序覆盖，但不泄漏评分器实现
- D3 在 `main.md` 中按表 3.8 的顺序覆盖：模块划分、代码复用、命名规范、
  注释质量、常量使用、复杂度控制。这样可以保证提示词与论文维度一一对应。
- 但措辞只使用**开发语言**，不写 AST、阈值、分值、权重或检测规则。否则会把
  D3 从**测代码质量**污染成**测是否迎合评分器实现**。
- D4 仍只保留一句泛化体验引导：为关键状态提供可见反馈。它不展开成指标清单，
  避免把体验维度变成另一组显式指令遵循测试。
- 理论依据见 RELATED_WORK.md（IFEval 的指令遵循构念 + 构念污染论证）。

## 三、架构：单一真源
- 每游戏 spec 的 `checkpoints` 既喂 prompt（告诉模型做什么），
  又喂 `dimension2` 评估器（作为打分靶子）。
- 这两处只能有一份数据，从同一个文件读。分成两份 → 出现
  "prompt 让做 A、评估器按 B 打分"的漂移。

## 四、不变量：可归因的前提
- main.md 骨架对所有模型、同难度所有游戏逐字节一致，只有 spec 数据变。
- 否则 prompt 差异与 model/game 差异混淆，分数无法归因到模型能力。
- 依据：Sclar et al. 2024。

## 一句话总结
**功能维显式写全，质量维按顺序覆盖但不泄漏实现，checkpoint 单一真源，骨架保持不变量。**
